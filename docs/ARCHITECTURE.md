## Component Details

### 1. API Gateway (FastAPI)
**File:** `api/server.py`

**Responsibilities:**
- REST API endpoints
- WebSocket connections
- Request validation (Pydantic)
- Authentication/Authorization
- Error handling
- CORS management

**Key Endpoints:**
- `POST /triage` - Classify patient
- `GET /dashboard` - Get dashboard data
- `GET /surveillance` - Get surveillance data
- `WebSocket /ws` - Real-time updates
- `POST /auth/login` - Authentication

### 2. Triage Agent (LangChain)
**File:** `langchain_integration/triage_agent.py`

**Architecture Pattern:** ReAct Agent

**Tools Available:**
1. **TriageTool** - Patient classification
2. **DashboardUpdateTool** - Update dashboard
3. **SurveillanceTool** - Outbreak analysis

**Flow:**
```
Patient Data → Agent → Tools → Results → Agent → Response
```

### 3. RAG System
**File:** `rag/rag_system.py`

**Components:**
- **TriageLLM** - Custom LLM wrapper
- **RAGSystem** - Main RAG orchestrator

**Classification Flow:**
```
1. Create query from patient data
2. Search vector database for similar cases
3. If cases found + LLM available → RAG classification
4. If no cases but LLM available → LLM-only
5. Otherwise → Rule-based + similarity weighting
```

### 4. Vector Database (ChromaDB)
**Purpose:** Store embeddings of historical cases

**Structure:**
- Collection: `mimic_cases`
- Embeddings: Generated by sentence-transformers
- Metadata: Patient info, ESI levels, vitals
- Similarity: Cosine similarity search

### 5. Surveillance System
**File:** `surveillance/syndromic_surveillance.py`

**Algorithms:**
- **TF-IDF** - Text vectorization
- **DBSCAN** - Clustering similar cases
- **Statistical Analysis** - Outbreak detection

### 6. Data Processing
**Files:**
- `data_processing/mimic_processor.py` - MIMIC data handling
- `data_processing/synthetic_data_generator.py` - Synthetic data

## Data Flow Diagrams

### Patient Classification Flow
```
1. Patient Data Input
   ↓
2. API Server (Validation)
   ↓
3. TriageAgent.process_patient()
   ↓
4. RAGSystem.classify_patient()
   ├─→ Create Query
   ├─→ Vector DB Search
   ├─→ Retrieve Similar Cases
   ├─→ LLM Classification (with context)
   └─→ Return Result
   ↓
5. Dashboard Update
   ↓
6. Surveillance Analysis
   ↓
7. Response to Client
```

### Real-Time Update Flow
```
1. Patient Classified
   ↓
2. WebSocket Broadcast
   ↓
3. All Connected Clients Notified
   ↓
4. Dashboard Updates Automatically
```

## Design Patterns Used

1. **Singleton Pattern** - Model caching, Agent instance
2. **Factory Pattern** - Tool creation
3. **Strategy Pattern** - Classification methods (RAG/LLM/Rule-based)
4. **Observer Pattern** - WebSocket broadcasting
5. **Repository Pattern** - Data access layer

## Performance Architecture

### Caching Layers
1. **Model Cache** - LLM model in memory
2. **Request Cache** - Classification results (Redis/Memory)
3. **Vector DB Cache** - Embeddings cached

### Optimization Strategies
- Lazy loading (agent loads on first request)
- Connection pooling (database)
- Batch processing (vector operations)
- Async operations (WebSocket)

## Scalability Considerations

### Current Limitations
- SQLite (single-writer)
- In-memory agent state
- Single model instance

### Future Scalability
- PostgreSQL migration
- Redis for shared state
- Model serving (separate service)
- Horizontal scaling (multiple API instances)

---

This architecture supports the core functionality while remaining extensible for future enhancements.
